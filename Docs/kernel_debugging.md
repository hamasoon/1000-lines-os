# 1. Looking at the disassembly of the kernel

run `llvm-objdump -d kernel.elf` to look at the disassembly of the kernel. You should see something like this:

```asm

80200004: 13 05 45 05  	addi	a0, a0, 84
80200008: 2a 81        	mv	sp, a0
8020000a: 6f 00 a0 01  	j	0x80200024 <kernel_main>
8020000e: 00 00        	unimp	

80200010 <memset>:
80200010: 09 ca        	beqz	a2, 0x80200022 <memset+0x12>
80200012: aa 86        	mv	a3, a0
80200014: 7d 16        	addi	a2, a2, -1
80200016: 13 87 16 00  	addi	a4, a3, 1

80200028: 13 05 45 05  	addi	a0, a0, 84

80200030: 93 85 45 05  	addi	a1, a1, 84
80200034: 33 86 a5 40  	sub	a2, a1, a0
80200038: 01 ce        	beqz	a2, 0x80200050 <kernel_main+0x2c>
8020003a: b3 05 b5 40  	sub	a1, a0, a1
8020003e: 2e 86        	mv	a2, a1
80200040: 93 06 15 00  	addi	a3, a0, 1
80200044: 85 05        	addi	a1, a1, 1
80200046: 23 00 05 00  	sb	zero, 0(a0)
8020004a: 36 85        	mv	a0, a3
8020004c: e3 f9 c5 fe  	bgeu	a1, a2, 0x8020003e <kernel_main+0x1a>
80200050: 01 a0        	j	0x80200050 <kernel_main+0x2c>
```

- you can see the `boot` function at address `0x80200000`, which sets up the stack pointer and jumps to `kernel_main`.\
- at `0x8020000a` inside the `boot` function, there is a jump instruction that jumps to `kernel_main` at address `0x80200024`.
    - this is the entry point of the kernel where it starts executing after the bootloader has loaded it into memory.
- the `memset` function is implemented to clear the BSS section, and `kernel_main` is the main entry point of the kernel where it initializes the BSS section and enters an infinite loop.
    - infinite loop is implemented in `0x80200050` where it jumps to itself, effectively halting the kernel execution there.

Interesting things to note:
- the `.text` section starts at address `0x80200000` and contains the `boot`, `memset`, and `kernel_main` functions.
- and no `.data` and `.rodata` sections are present in the kernel, which means there are no initialized global variables or read-only data in the kernel.
- the `.bss` section starts at address `0x80200054` and contains the uninitialized data, which is cleared to zero in `kernel_main`.
- the stack is allocated at the end of the `.bss` section, starting at address `0x80220054` and is defined by the symbol `__stack_top`.

# 3. Looking at register status of the kernel using QEMU
if you input `ctrl+c` in the QEMU terminal, you can enter to the QEMU monitor, where you can see a lot of debugging information about the virtual machine, including the register status of the CPU. You can use the `info registers` command to see the current values of all the registers. You should see something like this:

 - currently, this information is not very useful, but it can be helpful when we start implementing more complex features in the kernel, such as system calls, interrupts, and context switching, where we need to manipulate the registers to save and restore the state of the CPU.
